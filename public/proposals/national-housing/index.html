Overview

Ellen has developed a suite of property analysis and reporting tools that give her clients (typically nonprofit housing operators) clear visibility into the financial and operational health of their properties. The apps pull together financials, audit history, cash distributions, and backup documents into a single, organized view.

Right now these tools live locally and are shared via screen share or PDF printout. The goal of this engagement is to take what Ellen has built and turn it into a secure, stable, production web application that clients can log into directly from anywhere.

This proposal covers the initial build, ongoing support, and the terms of the engagement.

02

The problem we are solving

Ellen has proven the concept. Her clients are stunned by the clarity these dashboards provide. The gap is distribution and stability:

  • The apps run locally on her machine, not on the web
  • Clients can only access them via screen share or printed PDF
  • There is no login system, so clients cannot self-serve
  • There is no versioning or staging environment, so changes can break things unexpectedly
  • All files live in iCloud, which is not designed for managing code

The fix is moving from a vibe-coded prototype to a production application built on the developer superpower stack: Claude as the brain, GitHub for code management, Vercel for stable hosting, and a Postgres database for secure, persistent data.

03

Scope of work

The initial build covers three layers of the application.

3.1Login and user management

  • Secure login for Ellen (administrator) and her clients (read-only viewers)
  • Each client accesses only their own property data
  • Row-level security so no client can see another client's information
  • Password reset and basic account management

3.2Administrator tools

  • Ellen's admin dashboard to manage clients and properties
  • File and document upload interface so Ellen can update property data without needing Claude Code
  • Ability to attach backup documents (audits, financials, invoices) to each property
  • Controls to publish or hide information from client view

3.3Client-facing application

  • Clean, non-technical interface based on Ellen's existing prototypes
  • Dashboard views for financials, cash distributions, and audit history
  • Document library where clients can open and download attached files
  • View-only access; clients cannot modify data
  • PDF export so clients can print or save their reports

The codebase will be migrated from iCloud into GitHub, connected to Vercel for hosting, and backed by a Supabase (Postgres) database. This is the same stack we teach at our retreats and the one Ellen's son will recognize from his Stanford background.

How Human Tokens work

Human Tokens are the ongoing support model. Think of them like a bank of 40 engineering hours each month. Need a new feature? Submit it. Found a bug? We fix it. Want to improve the dashboard for a new client type? We build it. Flat rate, no nickel-and-diming, no hourly invoices.

Once the app is wired up and stable, many small changes can be made directly through Claude. The Human Token retainer covers the things Claude cannot do alone: anything that requires an engineer to review architecture, deploy changes safely, or build something new from scratch.
